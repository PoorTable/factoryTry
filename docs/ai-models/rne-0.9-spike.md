# Spike: react-native-executorch 0.9.x compatibility + hook API contract (APP-43)

De-risks the unmock chain (APP-40 → APP-41 → APP-42) before the upgrade lands.
Every claim below is backed by a cited source: a resolved npm `peerDependencies`
range, a `.d.ts` type signature copied from the published package, or a doc URL.

## TL;DR

| Question | Verdict |
| --- | --- |
| **Q1 Toolchain** | **GO** — pin `react-native-executorch@0.9.2`. peerDependencies are wildcards (`react: "*"`, `react-native: "*"`), so there is no npm-level conflict with our `react@19.2.3` / `react-native@0.85.3`. One caveat: RN 0.85 is on the SWM compatibility table, but the table does not yet list **Expo SDK 56** (tops out at 55). |
| **Q2 Hook API** | `useLLM` returns both a one-shot `generate(messages)` and a context-managed `sendMessage(text)`; both resolve to the final `string` AND stream into `response`. `useImageEmbeddings().forward(input: string \| PixelData)` accepts a file URI directly — no manual pixel decode. |
| **Q3 CLIP text side** | **Reachable.** `useTextEmbeddings` + model id `clip-vit-base-patch32-text` (constant `CLIP_VIT_BASE_PATCH32_TEXT`) gives the CLIP text encoder in the SAME CLIP space as `clip-vit-base-patch32-image`. The zero-shot design in `src/services/ai/identify/clip.ts` stands as-is. |

- Stable 0.9.x line published to npm: **0.9.0, 0.9.1, 0.9.2** (the rest of `0.9.x`
  is `0.9.0-nightly-*`). Latest stable = **0.9.2**.
  Source: `npm view react-native-executorch versions --json`.
- Typings were read from the **published 0.9.2 tarball** (the version installed in

  `npm pack react-native-executorch@0.9.2` →

## Q1 — Toolchain compatibility (peerDependencies / release)

**Target stack:** `react@19.2.3`, `react-native@0.85.3`, `expo@~56.0.8`.

**Resolved `peerDependencies` (cited):**

```text
$ npm view react-native-executorch@0.9.2 peerDependencies
{ react: '*', 'react-native': '*' }

$ npm view react-native-executorch@0.9.1 peerDependencies
{ react: '*', 'react-native': '*' }

$ npm view react-native-executorch@0.9.0 peerDependencies
{ react: '*', 'react-native': '*' }
```

Because the peer ranges are wildcards, **npm/yarn will not raise a peer conflict**
for our `react@19.2.3` / `react-native@0.85.3`. RNE does not pin `react` or
`react-native` at all — and it declares no `expo` peer dependency, so Expo SDK 56
does not trip an install-time conflict either.

**Independent confirmation from the SWM compatibility table:**
<https://docs.swmansion.com/react-native-executorch/docs/other/compatibility>

- RNE **0.9.x supports React Native 0.81–0.85** (0.85 marked "yes"). Our
  `react-native@0.85.3` is in range. ✅
- The same table lists **Expo SDK up to 55** for 0.9.x; **Expo SDK 56 is not yet
  listed**. This is a *documentation gap*, not a declared incompatibility (there is
  no Expo peer to violate, and SDK 56 is built on RN 0.85, which 0.9.x supports).

**Verdict: GO. Pin `react-native-executorch@0.9.2`** (latest stable 0.9.x).

- Reasoning for the pin: 0.9.2 is the newest stable in the 0.9 line and carries the
  `useImageEmbeddings` + CLIP text-encoder surface APP-42/APP-29 depend on (absent
  in our currently-installed 0.4.10).
- **Caveat to carry into APP-40:** verify the native build on Expo SDK 56 once,
  since the compat table predates SDK 56. RN 0.85 support is the load-bearing fact
  (SDK 56 = RN 0.85), and it is present, so risk is low.
- **Fallback (only if a SDK-56 native build issue surfaces):** the same hook API
  exists in **0.8.x** (the table shows identical RN 0.81–0.85 support for 0.8.x);
  pin `react-native-executorch@0.8.4` as a drop-back. No RN bump is required —
  our `react-native@0.85.3` is already within both lines' supported range.

## Q2 — Hook API contract (signatures copied from 0.9.2 typings)

### `useLLM` — turn generation + reading the final string

`package/lib/typescript/hooks/natural_language_processing/useLLM.d.ts`:

```ts
export declare function useLLM<C extends readonly LLMCapability[]>(props: LLMProps & {
    model: { capabilities: C };
}): LLMTypeMultimodal<C>;
export declare function useLLM(props: LLMProps): LLMType;
```

Return shape — `package/lib/typescript/types/llm.d.ts` (`LLMTypeBase` + `LLMType`):

```ts
export interface LLMTypeBase {
    /** Updated after model responds to sendMessage. */
    messageHistory: Message[];
    /** Updated with each token generated by the model (streamed). */
    response: string;
    token: string;
    isReady: boolean;
    isGenerating: boolean;
    downloadProgress: number;
    error: RnExecutorchError | null;
    configure: ({ chatConfig, toolsConfig, generationConfig }: LLMConfig) => void;
    /**
     * Runs model to complete chat passed in `messages`. Does NOT manage context.
     * @returns The generated tokens as `string`.
     */
    generate: (messages: Message[], tools?: LLMTool[]) => Promise<string>;
    interrupt: () => void;
    // ...token-count helpers, deleteMessage(index)
}

export interface LLMType extends LLMTypeBase {
    /** Adds a user message to the conversation; updates messageHistory.
     *  @returns The model's response as a `string`. */
    sendMessage: (message: string) => Promise<string>;
}
```

**Contract for APP-42's adapter:**

- **Two ways to drive a turn:**
  1. `sendMessage(text)` — context-managed; appends to `messageHistory`, returns the
     final assistant string. Use for the multi-turn coach.
  2. `generate(messages, tools?)` — stateless one-shot over an explicit `Message[]`;
     returns the final string. Use for single-shot structured prompts (palette /
     outfit JSON) where you build the message array yourself.
- **Reading the final string — BOTH are supported:**
  - **Return value:** `await sendMessage(...)` / `await generate(...)` resolves to the
    full `string`. This is the simplest path for a non-streaming adapter.
  - **Streamed field:** `response` accumulates token-by-token (and `token` is the
    latest token) while `isGenerating` is true — for live UI streaming.
- A turn is *one* call (`sendMessage`/`generate`), not a separate "start + poll".
  `Message = { role: 'user'|'assistant'|'system'; content: string; mediaPath?; ... }`.

### `useImageEmbeddings.forward` — accepts a file URI

`package/lib/typescript/hooks/computer_vision/useImageEmbeddings.d.ts`:

```ts
export declare const useImageEmbeddings: ({ model, preventLoad }: ImageEmbeddingsProps) => ImageEmbeddingsType;
```

`package/lib/typescript/types/imageEmbeddings.d.ts`:

```ts
export type ImageEmbeddingsModelName =
  | 'clip-vit-base-patch32-image'
  | 'clip-vit-base-patch32-image-quantized';

export interface ImageEmbeddingsType {
    error: RnExecutorchError | null;
    isReady: boolean;
    isGenerating: boolean;
    downloadProgress: number;
    /**
     * Supports two input types:
     *  1. String path/URI: File path, URL, or Base64-encoded string
     *  2. PixelData: raw pixel data from image libraries
     * @returns Promise<Float32Array> embedding vector.
     */
    forward: (input: string | PixelData) => Promise<Float32Array>;
    runOnFrame: ((frame: Frame) => Float32Array) | null;
}
```

**Confirmed:** `forward` accepts `string | PixelData`, and the doc comment states the
string may be a **"File path, URL, or Base64-encoded string"** — so the camera photo
URI is fed directly, **no manual pixel decode required**. (`runOnFrame` is the
VisionCamera-worklet variant; not needed for the still-photo identify flow.)

## Q3 — CLIP text side (zero-shot needs image AND text embeddings in one CLIP space)

`src/services/ai/identify/clip.ts` does zero-shot classification: it embeds the
photo once (`embedder.encodeImage`) and embeds each candidate label prompt
(`embedder.encodeText`), then takes cosine similarity. That **requires a text
encoder in the same CLIP space as the image encoder** — see the `ClipEmbedder`
interface it depends on:

```ts
// src/services/ai/identify/clip.ts
export interface ClipEmbedder {
  encodeImage: (buffer: ArrayLike<number>, channels: PixelChannels) => Promise<number[]>;
  encodeText: (prompt: string) => Promise<number[]>;
}
```

**The CLIP text encoder is reachable in 0.9.2.** It is exposed via `useTextEmbeddings`
with the dedicated CLIP-text model id.

Model name union — `package/lib/typescript/types/textEmbeddings.d.ts`:

```ts
export type TextEmbeddingsModelName =
  | 'all-minilm-l6-v2'
  | 'all-mpnet-base-v2'
  | 'multi-qa-minilm-l6-cos-v1'
  | 'multi-qa-mpnet-base-dot-v1'
  | 'distiluse-base-multilingual-cased-v2-8da4w'
  | 'paraphrase-multilingual-minilm-l12-v2-quantized'
  | 'clip-vit-base-patch32-text';   // <-- CLIP text encoder

export interface TextEmbeddingsType {
    // ...
    forward(input: string): Promise<Float32Array>;
}
```

Constant — `package/lib/typescript/constants/modelUrls.d.ts` (image + text are the
**same CLIP checkpoint**, `software-mansion/react-native-executorch-clip-vit-base-patch32`):

```ts
export declare const CLIP_VIT_BASE_PATCH32_IMAGE: {
    readonly modelName: "clip-vit-base-patch32-image";
    readonly modelSource: "https://huggingface.co/software-mansion/react-native-executorch-clip-vit-base-patch32/resolve/v0.9.0/xnnpack/clip_vit_base_patch32_image_xnnpack_fp32.pte";
};
export declare const CLIP_VIT_BASE_PATCH32_TEXT: {
    readonly modelName: "clip-vit-base-patch32-text";
    readonly modelSource: "https://huggingface.co/software-mansion/react-native-executorch-clip-vit-base-patch32/resolve/v0.9.0/xnnpack/clip_vit_base_patch32_text_xnnpack_fp32.pte";
    readonly tokenizerSource: "https://huggingface.co/software-mansion/react-native-executorch-clip-vit-base-patch32/resolve/v0.9.0/tokenizer.json";
};
```

**Concrete path for APP-29 / `clip.ts`:**

- `encodeImage` ← `useImageEmbeddings({ model: CLIP_VIT_BASE_PATCH32_IMAGE })`,
  `.forward(photoUri)` → `Float32Array`.
- `encodeText` ← `useTextEmbeddings({ model: CLIP_VIT_BASE_PATCH32_TEXT })`,
  `.forward(prompt)` → `Float32Array`.
- Both halves of the same `clip-vit-base-patch32` checkpoint ⇒ shared embedding
  space ⇒ the cosine/softmax zero-shot math in `clip.ts` is valid unchanged.

**Design impact on `clip.ts`: none structural.** Two adjustments for the adapter
(APP-42), not the pure pipeline:

1. `ClipEmbedder` currently types embeddings as `number[]`; the runtime returns
   `Float32Array`. The adapter wrapping the two hooks should convert
   (`Array.from(float32)`) before handing off to `clip.ts`, or `clip.ts`'s
   `cosine`/`softmax` can be widened to `ArrayLike<number>` (they already index
   numerically, so the change is trivial).
2. `encodeImage` in `clip.ts` takes `(buffer, channels)`, but `useImageEmbeddings`
   wants a **URI/PixelData**. The adapter should call the hook with the photo URI
   directly (Q2 confirms URIs are accepted), making the manual pixel buffer that
   `clip.ts` threads through largely vestigial for the CLIP image step — the
   swatch/color extraction still uses the pixel buffer, so keep both inputs at the
   call site.

Neither point invalidates the zero-shot approach; both are wiring details for the
APP-42 adapter.

## Linear follow-ups (APP-40 / APP-41 / APP-42) — G8

**No assumptions were invalidated.** All three spike questions returned the
*positive* outcome the unmock chain assumed:

- **APP-40 (upgrade):** assumed 0.9.x is installable on our stack — **confirmed**
  (wildcard peers, RN 0.85 supported). The only addition is a low-risk verification
  note (build once on Expo SDK 56, since the compat table predates it) and the pin
  recommendation `0.9.2` with `0.8.4` as fallback. This is guidance, not a
  contradiction of an APP-40 assumption.
- **APP-41 / APP-42 (adapter):** assumed `useLLM` exposes a single-call turn that
  yields the final string and `useImageEmbeddings.forward` takes a URI —
  **both confirmed** from the typings above.
- **APP-29 (`clip.ts`):** assumed an in-space CLIP text encoder exists —
  **confirmed** (`clip-vit-base-patch32-text`). Only minor adapter-side wiring
  (`Float32Array`→`number[]`, URI-based image embedding) is needed, which lives in
  APP-42's adapter, not in the pure `clip.ts` pipeline.

Because nothing was invalidated, no corrective Linear comments are required by the
ticket's acceptance criterion (comments are mandated only *when a finding
invalidates an assumption*). A confirming note can optionally be posted to
APP-40/APP-41/APP-42 to share the exact pin (`0.9.2`) and the SDK-56 verification
caveat.
